# AGENTS.md

## Project purpose

This repository is a configuration layer for the Cline AI agent in VSCode.
It encodes the standards, conventions, and reusable knowledge required to develop
professional Next.js 15+ applications. The repository itself contains no application
code — only rules, skills, workflows, and templates that Cline loads when assisting
a developer.

When this repository is present in a project, Cline behaves as a senior fullstack
engineer specializing in Next.js 15 App Router, React 19, TypeScript strict mode,
Tailwind CSS, and shadcn/ui. All rules are written for that stack.

## How rules work

Rules in `.clinerules/` are always active. They are not invoked — they form the
baseline behavior for every response. When working in any project that uses this
configuration, apply all rules simultaneously.

Priority order when rules conflict:
1. `30-security.md` — security constraints are never relaxed
2. `20-quality-gates.md` — TypeScript and linting rules
3. `10-nextjs-conventions.md` — framework-specific patterns
4. `00-context.md` — architecture and naming

---

## How skills work

Skills in `.cline/skills/` are loaded on demand when the user's request matches
the `description` field in `SKILL.md`. They are not active by default and do not
consume context when unused.

Each skill directory can contain:
- `SKILL.md` — trigger description + actionable instructions for the agent
- `docs/` — detailed reference documents linked from SKILL.md
- `templates/` — ready-to-use file templates
- `scripts/` — automation scripts

Implemented skills:

| Skill | Triggers on |
|---|---|
| `mdx-blog` | Creating blog posts, MDX setup, gray-matter, remark/rehype plugins, frontmatter |
| `shadcn-component` | Installing shadcn components, theming, cn(), variants, wrappers |
| `zod-schema` | Validation schemas, form data, search params, Server Action input, env vars |

Stub skills (SKILL.md is empty, not yet usable):

| Skill | Intended purpose |
|---|---|
| `nextjs-app-router` | App Router routing patterns, error handling, page templates |
| `nextjs-pr-review` | Pull request review checklist for Next.js projects |

## Architecture the rules enforce

Target project structure:

```
src/
├── app/                    # Routing only — layout, page, loading, error, route files
├── features/[name]/        # One directory per business domain
│   ├── components/
│   ├── hooks/
│   ├── actions.ts          # Server Actions
│   ├── queries.ts          # Data fetching (server-only)
│   └── types.ts            # Zod schemas + inferred types
├── components/ui/          # shadcn/ui components (generated by CLI, never edited directly)
├── components/             # Shared custom components and wrappers
├── lib/                    # Utilities: cn(), dal.ts, env.ts, posts.ts, etc.
└── types/                  # Global TypeScript types and enums
```

Key constraints from `00-context.md`:
- Server Components by default; `'use client'` only when interactivity requires it
- Server Actions for mutations; Route Handlers for public-facing APIs
- No ORM, no database — this configuration targets filesystem-based or API-driven projects
- Strict TypeScript: no `any`, no `@ts-ignore`, no untyped exports
- All imports use `@/*` path alias — no relative imports that escape feature boundaries

## Security model (`30-security.md`)

The most critical rule: **middleware is not a security boundary**.

The CVE-2025-29927 vulnerability (Next.js 11.1.4–15.2.2) allowed bypassing middleware
auth via the `x-middleware-subrequest` header. Fix: Next.js 15.2.3+. The correct pattern
is a Data Access Layer (DAL) at `src/lib/dal.ts` that:

1. Starts with `import 'server-only'`
2. Calls `verifySession()` before every data operation
3. Returns DTO objects, never raw ORM results
4. Checks resource ownership (prevents IDOR)

Server Action security sequence: authenticate → validate (Zod safeParse) → authorize → mutate → revalidate cache.

Environment variables: secrets never use `NEXT_PUBLIC_` prefix; `.env.local` is always
in `.gitignore`; only `.env.example` is committed.

## Quality gates (`20-quality-gates.md`)

TypeScript `tsconfig.json` must include beyond `strict: true`:

```json
"noUncheckedIndexedAccess": true,
"exactOptionalPropertyTypes": true,
"noImplicitReturns": true,
"noUnusedLocals": true,
"noUnusedParameters": true,
"verbatimModuleSyntax": true
```

Prettier settings: `semi: false`, `singleQuote: true`, `tabWidth: 4`,
`trailingComma: "all"`, `printWidth: 100`, with `prettier-plugin-tailwindcss`.

Pre-commit (husky + lint-staged): ESLint --fix and Prettier --write run automatically.
Before every commit, also run manually: `tsc --noEmit`, `eslint .`, `next build`.

## Next.js 15 breaking changes (`10-nextjs-conventions.md`)

The most common source of bugs when migrating from Next.js 14:

- `params` and `searchParams` props are now `Promise<...>` — must `await` them in
  Server Components, or use `React.use()` in Client Components
- `fetch()` is not cached by default — specify `{ cache: 'force-cache' }` or
  `{ next: { revalidate: n } }` explicitly
- `next/dynamic` with `ssr: false` cannot be used directly in Server Components —
  wrap in a Client Component
- Middleware runs on Edge Runtime by default; Node.js runtime is stable from 15.5+

## Working with this repository

When modifying rules or skills:

- Rules in `.clinerules/` use number prefixes to control load order (00 < 10 < 20 < 30)
- Skill `description` fields in SKILL.md frontmatter determine when the skill activates;
  write them in third person, include specific trigger phrases the user might say
- Keep each SKILL.md under ~3500 words; move details to `docs/` with markdown links
- Workflows in `.clinerules/workflows/` are currently stubs — implement as step-by-step
  procedures using the same format as the filled-in rules files
- The `nextjs-app-router` and `nextjs-pr-review` skills are stubs — SKILL.md files
  exist but are empty

Commit convention (enforced by rules): `feat:`, `fix:`, `docs:`, `refactor:`,
`chore:`, `perf:` prefixes. No `console.log` in committed code.
